# AI 修改记录日志

> 用于记录 AI 参与的代码/文档修改，便于后续迭代、回溯与复盘。

## 记录规范

1. 写下时间戳。  
2. 简要记录查询目的、输入内容和分析结果。
3. 变更摘要描述：
   - 需包含“变更内容”、“变更原因”与“影响范围”，在“变更摘要”中分点说明。
   - 变更内容应简明扼要，突出核心修改点。  
   - 变更原因应说明为何需要该修改（如修复 bug、优化性能、添加功能等）。  
   - 影响范围应评估该修改对项目的潜在影响（如仅文档、代码逻辑、用户体验等）。
4. 若有风险或待办，请写入“后续迭代建议”。如果有其他的需要注意的事项，也可以写入“修改注意事项”中。
5. 记录格式应保持一致，便于后续统计与分析。

## 记录规范追加说明

1. 记录分类：
   - 更改：仅记录已实际发生或即将落地的代码/文档修改。
   - 分析：所有未落地的讨论、设计、推荐、解释等。

2. 更改记录必须包含：
   - 变更类型（feat / fix / refactor / docs / perf / experiment）
   - 变更内容 / 变更原因 / 影响范围

3. 分析记录建议包含：
   - 输入内容
   - 分析结果
   - 结论级别（草案 / 可执行 / 已验证）

4. 记录顺序：
   - 采用倒序（最新在上）
   - 编号递增（保证唯一性）

5. 目标原则：
   - 可追溯（能还原上下文）
   - 可审计（区分分析与落地）
   - 可复用（能指导后续工作）

```
变更文件：（列出修改涉及的文件路径，如 `src/module/file.py`）
变更摘要：
- 变更内容：（简要描述修改的具体内容，如“修复了某个函数的边界条件错误”、“优化了某个算法的时间复杂度”、“添加了一个新的 API 接口”等）
- 变更原因：（说明为什么需要进行该修改，如“该修改修复了一个已知的 bug”、“该修改提升了系统性能”、“该修改满足了新的需求”等）
- 影响范围：（评估该修改对项目的潜在影响范围，如“该修改仅影响文档内容”、“该修改可能影响相关模块的功能”、“该修改可能引入新的依赖”等）
后续迭代建议：（如果有风险或待办事项，请在此处写入，如“需要在后续版本中进行回归测试”、“需要在后续版本中优化相关功能”、“需要在后续版本中添加相关文档说明”等）
修改注意事项：（如果有其他需要注意的事项，请在此处写入，如“该修改涉及到敏感代码，请谨慎评审”、“该修改可能引入新的依赖，请确保相关环境配置正确”等）
```
